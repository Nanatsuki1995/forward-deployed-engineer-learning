# AI 协作开发规范

## Universal AI Development Loop

本协议定义一套可迁移到任何项目的 AI 辅助开发循环。目标不是让每个任务都变重，而是让 AI 在不同仓库中稳定做到：先理解现状，再最小修改，再真实验证，最后交付可审计结果。

`AGENTS.md` 是本规范的源文件；`CLAUDE.md` 是给 Claude 读取的同步副本。修改任一文件时，必须同步另一个文件，避免不同 AI 工具读取到不一致的规则。

## Core Rules（核心原则）

- **先读后改**：先读取项目配置、相关代码、现有约定，再动手修改。
- **保护用户改动**：不覆盖、不回滚、不格式化与本次任务无关的已有变更。
- **命令从项目发现**：只能使用项目已有脚本、配置和工具链；没有的命令跳过，不为了通过流程强行引入。
- **小步可验证**：优先做最小、局部、可验证的增量修改。
- **失败要读日志**：任何失败都先读懂错误输出，不能盲目重试。
- **交付要有证据**：最终说明改了什么、跑了什么、结果如何、什么没跑以及原因。

## Phase -1: Classify（任务分级）

**目标**：根据用户请求选择合适强度的 loop，避免简单任务过度工程化，也避免高风险任务验证不足。

先判断任务类型：

| 类型 | 示例 | 行为要求 |
|------|------|----------|
| Answer-only | 解释代码、回答概念、给建议 | 只读必要上下文，不改文件，不跑完整门禁 |
| Read-only review | 代码审查、方案评估、风险分析 | 只读和分析，除非用户明确要求修复 |
| Small edit | 文案、小样式、单文件小修 | 修改最小范围，运行受影响的最小检查 |
| Bugfix | 修复报错、异常行为、测试失败 | 先复现或定位根因，再修复，再跑相关回归 |
| Feature | 新功能、新接口、新页面 | 完整执行 Orient → Plan → Execute → Verify → Review |
| Release / Deploy | 发布、合并、部署、打包 | 增加版本、变更摘要、回滚点和发布后验证 |
| Destructive / Config / Dependency | 删除数据、改依赖、改构建、改部署 | 先说明影响和风险，获得明确确认后再执行 |

任务升级规则：

- 如果一个 Small edit 触及公共 API、共享组件、数据库、认证、权限、构建配置或依赖声明，升级为 Feature / Bugfix。
- 如果用户明确要求“实现 / 修复 / 提交 / 测试 / 发布”，不要停留在建议层；在安全范围内完成端到端工作。
- 如果用户明确要求“只看 / 只评审 / 不修改”，保持只读。

## Phase 0: Orient（定位）

**目标**：建立对当前状态、项目工具链和影响范围的共同理解。

### 0.1 检查工作区

开始任何修改前，先检查当前工作区：

- 运行 `git status --short`。
- 识别已修改、已暂存、未跟踪文件。
- 不回滚用户已有改动。
- 修改一个已经有变更的文件前，必须先读取当前内容，并在现有状态上继续。
- 如果发现与任务相关但来源不明的改动，先理解它；只有在无法安全继续时才向用户确认。

### 0.2 读取项目指令

优先读取项目内的协作说明和规范文件，例如：

- `AGENTS.md`
- `CLAUDE.md`
- `README.md`
- `CONTRIBUTING.md`
- `.editorconfig`
- 项目特有的 docs、architecture、rules 或 skill 文件

如果多个说明冲突，优先级为：用户最新请求 → 当前会话指令 → 仓库内更具体路径的说明 → 通用说明。

### 0.3 发现项目工具链

从项目配置文件中发现命令和工具，不凭记忆猜测：

- JavaScript / TypeScript：`package.json`、`pnpm-workspace.yaml`、`yarn.lock`、`package-lock.json`
- Python：`pyproject.toml`、`requirements.txt`、`uv.lock`、`poetry.lock`
- Rust：`Cargo.toml`
- Go：`go.mod`
- Java / Kotlin：`pom.xml`、`build.gradle`
- Make / Task：`Makefile`、`Taskfile.yml`
- Docker / Compose：`Dockerfile`、`docker-compose*.yml`
- CI：`.github/workflows/*`、`.gitlab-ci.yml`

发现后在心里形成本项目的命令矩阵：

| 项目能力 | 命令 | 来源 |
|----------|------|------|
| 安装依赖 | 项目已有命令 | 配置文件 |
| 启动开发服务 | 项目已有命令 | 配置文件 |
| Lint / Format | 项目已有命令 | 配置文件 |
| 类型检查 / 编译 | 项目已有命令 | 配置文件 |
| 构建 | 项目已有命令 | 配置文件 |
| 单元测试 | 项目已有命令 | 配置文件 |
| 集成 / E2E 测试 | 项目已有命令 | 配置文件 |
| 数据库 / 代码生成 | 项目已有命令 | 配置文件 |

如果某项能力不存在，记录为“不存在 / 不适用”，不要新增工具链来满足流程。

### 0.4 理解影响范围

修改前阅读：

- 将要修改的文件。
- 直接依赖和被依赖文件。
- 上游调用方。
- 下游消费者。
- 相关测试。
- 相关文档或 README。

对于大范围改动，先列出受影响文件和行为边界，再动手。

## Phase 1: Plan（计划）

**目标**：确定具体改什么、怎么验证，以及如何控制范围。

### 1.1 建立基线

根据任务类型选择基线：

- Answer-only / Read-only review：不需要运行测试，除非用户要求。
- Small edit：运行最小相关检查即可。
- Bugfix：优先运行能复现问题的命令或测试。
- Feature：运行项目已有的相关测试或全量基线。
- Release / Deploy：运行完整门禁，并确认发布前状态。

如果基线失败：

- 判断失败是否与本次任务相关。
- 不相关：记录为 existing baseline failure，不擅自扩大范围修复。
- 相关或阻塞：先定位根因；必要时修复或向用户确认范围。
- 如果项目长期没有稳定基线，使用受影响范围的最小验证作为临时基线，并在最终报告中说明。

### 1.2 明确变更边界

在实现前明确：

- 哪些文件会改。
- 哪些函数、接口、组件、命令或行为会变。
- 哪些行为必须保持不变。
- 是否需要同步测试、文档、类型、数据库迁移或示例。

对于新功能：

- 先确定接口、类型、数据流或组件边界。
- 再写实现。
- 最后补齐验证。

对于 Bug 修复：

- 先复现或解释为什么无法复现。
- 优先添加能覆盖该问题的测试。
- 再修复根因。

### 1.3 选择测试策略

按改动类型选择验证：

| 改动类型 | 推荐验证 |
|----------|----------|
| 纯文档 | 检查链接、命令、示例、diff |
| 工具脚本 | 最小输入样例 + 错误路径 |
| UI 组件 | 组件测试、页面渲染、浏览器控制台 |
| API | 成功、失败、权限、输入校验 |
| 数据库 | migration / schema validate / generate / seed 相关命令 |
| 认证 / 权限 | 正向权限、拒绝路径、越权路径 |
| 公共库 / SDK | 单元测试、类型检查、最小使用示例 |
| 构建配置 | clean build、相关 CI 命令、本地启动 |

## Phase 2: Execute（执行）

**目标**：按计划编写代码，保持改动局部、清晰、可回滚。

执行规则：

- 遵循项目现有命名、目录结构、错误处理、测试风格和注释密度。
- 优先使用现有 helper、组件、服务、类型和工具函数。
- 不引入新范式、新依赖、新构建工具，除非这正是用户要求的变更。
- 不做与任务无关的重构。
- 不把格式化扩散到无关文件。
- 不留下调试代码，例如 `console.log`、`debugger`、临时 mock、临时 print。
- 对复杂逻辑只写必要注释，说明意图，不复述代码。

依赖、构建、部署、数据库等高影响变更：

- 新增、删除或升级依赖前，先说明原因和影响。
- 修改构建、CI、Docker、部署配置前，先说明风险。
- 数据库迁移必须说明兼容性、回滚方式或不可逆点。

---

## Phase 3: Verify（验证）

**目标**：确认变更正确，并尽量证明没有破坏已有功能。

### 3.1 分层验证

开发中使用最小门禁，交付前使用最终门禁。

最小门禁：

- 只运行与受影响范围直接相关的 lint、typecheck、test 或 build。
- 适合小步迭代，减少无关等待。

最终门禁：

- 优先运行项目级 lint / format check。
- 再运行类型检查 / 编译。
- 再运行构建。
- 再运行单元测试 + 集成测试。
- 最后运行 E2E 或浏览器验证（如果存在且可行）。

默认顺序：

```text
lint / format → typecheck / compile → build → unit / integration test → e2e / manual verification
```

如果项目没有某个命令，跳过并说明“不存在 / 不适用”。

### 3.2 自动修改命令

如果 lint、format 或生成命令会自动修改文件：

- 运行后必须重新检查 `git diff`。
- 确认自动修改只属于本次任务。
- 如果自动修改扩大到无关文件，停止并重新评估。

### 3.3 错误日志处理

任何验证失败：

- 读取完整错误输出。
- 找到根因假设。
- 不盲目重复同一个命令。
- 不跳过错误直接汇报成功。
- 记录失败属于新问题、已有基线问题，还是环境问题。

---

## Phase 4: Iterate（迭代）

**目标**：根据验证结果修正代码，直到通过或达到升级条件。

迭代规则：

- 每次只针对明确根因做修复。
- 修复后重新运行失败的检查。
- 关键修复完成后重新运行完整相关门禁。
- 如果同一错误连续出现，停止机械重试，重新分析方案。

迭代上限：

- 同一错误连续出现 3 次：暂停，重新评估方案。
- 不同错误累计达到 7 次：暂停，向用户报告进展并请求方向。
- 连续 2 轮迭代没有任何进展：暂停，重新设计。
- 单轮修改超过 5 个文件：检查是否范围膨胀。

---

## Phase 5: Review（审阅）

**目标**：交付前自查，降低低级错误和无关改动。

### 5.1 Diff 自审

交付前必须检查 diff：

- `git diff`：查看工作区改动。
- `git diff --cached`：如果涉及提交，只看暂存区改动。
- `git status --short`：确认没有意外文件。

逐文件确认：

- 每一行改动都属于本次任务。
- 没有遗留调试代码。
- 没有无关格式化。
- 没有误删用户内容。
- 错误处理没有吞异常。
- 用户可见错误信息不暴露内部细节。
- 新增注释准确描述意图。

### 5.2 行为验证

如果项目有运行命令，且任务涉及可运行行为：

- API / 服务：用最小请求验证核心路径。
- UI：打开关键页面，确认可渲染、可交互、无控制台错误。
- CLI / 库：用最小示例验证命令或 API 行为。
- 文档：确认命令、路径、示例与当前代码一致。

### 5.3 文档同步

如果行为、命令、接口、环境变量或用户流程变化：

- 同步 README、docs、示例或 changelog。
- 不在实现未稳定前过早更新文档。
- 最终文档必须描述实际交付内容，而不是计划内容。

---

## Phase 6: Deliver（交付）

**目标**：让用户清楚知道结果、证据和剩余风险。

最终回复应包含：

- **改动摘要**：做了什么。
- **修改文件**：列出关键文件。
- **验证结果**：列出实际运行的命令和结果。
- **未运行项**：说明没有运行的检查及原因。
- **风险 / 后续**：只列真实存在的剩余风险或建议。

如果用户要求“测试结果”和“修改报告”，必须明确给出，不作为可选项。

如果用户要求提交：

- 先检查 `git status --short`。
- 只根据 `git diff --cached` 或明确要提交的文件写提交信息。
- 不把未确认的无关改动放进提交。
- 提交前完成验证和 diff 自审。

## Escalation Rules（升级规则）

以下情况暂停自动执行，向用户说明情况并请求确认：

| 触发条件 | 说明 |
|----------|------|
| 需要新增、删除或升级核心依赖 | 依赖变化会影响安装、构建、供应链和运行环境 |
| 需要修改构建、CI、Docker 或部署配置 | 可能影响发布和生产环境 |
| 需要执行破坏性命令 | 例如删除文件、大规模移动、清空数据、重置分支 |
| 需要数据库不可逆迁移 | 例如删除列、清表、重写历史数据 |
| 需要访问或修改生产环境 | 包括生产数据库、生产密钥、生产部署 |
| 同一编译 / 类型错误连续出现 3 次 | 当前方案可能有根本问题 |
| 测试中有 3 个以上不相关用例失败 | 可能破坏了无关功能或基线不稳定 |
| 单轮迭代修改超过 5 个文件 | 范围可能膨胀 |
| 连续 2 轮迭代无进展 | 当前方向可能无效 |
| 用户请求与仓库规则冲突 | 需要用户决定优先级 |

## Protocol Parameters（协议参数）

以下参数可根据项目或用户偏好调整：

| 参数 | 默认值 | 说明 |
|------|:------:|------|
| 最大连续相同错误 | 3 | 超过则暂停并重新评估 |
| 最大累计错误 | 7 | 超过则报告进展并请求方向 |
| 最大无进展轮次 | 2 | 超过则停止机械迭代 |
| 最大单轮修改文件数 | 5 | 超过则检查是否范围膨胀 |
| 默认质量门禁顺序 | lint → type → build → test → e2e | 由快到慢、由静态到行为 |
| 默认交付证据 | diff + 命令结果 + 未运行项 | 保证结果可审计 |

## Portable Project Adapter Template（项目适配模板）

把本规范复制到新项目时，不需要改动核心 loop。AI 每次进入项目后，应根据实际配置临时生成以下适配信息：

```md
Project:
- Name:
- Package manager:
- Runtime:
- Workspace layout:

Commands:
- Install:
- Dev:
- Lint:
- Format:
- Typecheck / Compile:
- Build:
- Unit test:
- Integration test:
- E2E:
- DB / Codegen:

Local rules:
- Architecture:
- Naming:
- Tests:
- Docs:
- Do-not-touch:
```

如果项目希望长期固化这些信息，可以在仓库中维护单独的 `PROJECT_ADAPTER.md`、README 章节或本文件末尾的 Local Overrides。核心 loop 不应因单个项目而变形。
