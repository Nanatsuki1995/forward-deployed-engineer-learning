# AI 协作开发规范

## Development Loop Protocol

本协议定义了 AI 辅助开发的标准化循环，适用于任何项目类型。

### Phase 0: Orient（定位）

**目标**：建立对当前状态和变更范围的共同理解。

1. **发现项目工具链**
   - 从项目配置文件（`package.json`、`Makefile`、`Cargo.toml`、`pyproject.toml` 等）中识别可用命令。
   - 确定：测试命令、构建命令、lint/格式化命令、类型检查命令、启动/运行命令。
   - 如果某个命令不存在（如项目没有 lint），跳过对应检查，不要强行引入。

2. **理解影响范围**
   - 阅读将要修改的文件及其直接依赖。
   - 识别受影响的上游调用方和下游消费者。
   - 对于大范围改动，先列出所有受影响文件再动手。

### Phase 1: Plan（计划）

**目标**：确定具体改什么、怎么改，并建立基线。

3. **建立基线**
   - 运行项目现有的测试集，确认当前状态为通过。
   - 如果基线失败，先修复已有问题再进入新功能。

4. **明确变更边界**
   - 定义本次变更的精确范围：哪个文件、哪个函数、什么行为变化。
   - 对于新功能，先写出接口/类型/函数签名，再写实现。
   - 对于 Bug 修复，先写出能复现 Bug 的测试用例。

### Phase 2: Execute（执行）

**目标**：按计划编写代码。

5. **遵循项目既有模式**
   - 模仿现有代码的命名习惯、目录结构、错误处理方式、注释密度。
   - 不在本次变更中引入新的范式、工具或模式，除非变更本身就是重构。
   - 优先做最小的、可验证的增量修改。

### Phase 3: Verify（验证）

**目标**：确认变更正确，且没有破坏任何已有功能。

6. **逐级运行质量门禁**（按顺序，前一步失败则停止并进入 Phase 4）
   - Lint / 格式化 — 确保代码风格一致。
   - 类型检查 / 编译 — 确保没有类型错误或编译失败。
   - 构建（如项目有 build 步骤）— 确保产物可生成。
   - 单元测试 + 集成测试 — 确保功能正确且无回归。
   - 端到端测试（如果存在）— 确保关键链路完整。

7. **阅读错误日志**
   - 任何一步失败，必须读取并理解完整的错误输出。
   - 禁止跳过错误直接重试，禁止向用户提问而不先分析日志。

### Phase 4: Iterate（迭代）

**目标**：根据验证结果修正代码，直至全部通过。

8. **根因修复**
   - 针对每个失败项，定位根本原因而非表象。
   - 每轮修复后重新运行 Phase 3 全部门禁。

9. **迭代上限**
   - 同一错误连续出现 3 次 → 暂停，重新评估方案后再试。
   - 不同错误累计达到 7 次 → 暂停，向用户报告进展并请求方向确认。
   - 单轮迭代中修改超过 5 个文件 → 检查是否范围膨胀。

### Phase 5: Review（审阅）

**目标**：在交付前进行自查，降低低级错误率。

10. **自审变更**
    - 用 `git diff` 逐文件审视改动，确认每一行都是有意为之：
      - 有无遗留的调试代码（console.log、debugger、print 等）？
      - 有无与变更无关的文件被意外修改？
      - 新增的注释是否准确描述了代码意图？
      - 错误处理是否完整（不吞异常、不给用户看内部错误）？

11. **行为验证**
    - 如果项目有运行/启动命令，在可行的情况下启动并手动验证核心行为。
    - 对于 UI 项目，确认关键页面可渲染且无控制台报错。
    - 对于 CLI/库项目，用最小示例验证 API 行为。

---

### Escalation Rules（升级规则）

以下情况暂停 Loop 并向用户求助，不继续自动迭代：

| 触发条件 | 说明 |
|---------|------|
| 同一编译/类型错误连续出现 3 次 | 方案可能有根本性问题 |
| 需要修改包管理器的依赖声明 | 新增/删除/升级核心依赖属于架构决策 |
| 需要修改构建或部署配置 | 如 webpack、vite、docker、CI 配置 |
| 测试中有 3 个以上不相关用例因本次改动而失败 | 改动可能破坏了不相关的功能 |
| 单轮迭代已修改超过 5 个文件 | 范围可能膨胀，需要确认 |
| 连续 2 轮迭代无任何进展 | 当前方案无效，需要重新设计 |

### 协议参数

以下参数可根据项目或用户偏好调整：

| 参数 | 默认值 | 说明 |
|------|:---:|------|
| 最大连续相同错误 | 3 | 超过则升级 |
| 最大累计错误 | 7 | 超过则升级 |
| 最大单轮修改文件数 | 5 | 超过则检查范围 |
| 最大无进展轮次 | 2 | 超过则升级 |
| 质量门禁顺序 | lint → type → build → test → e2e | 由快到慢排列 |
